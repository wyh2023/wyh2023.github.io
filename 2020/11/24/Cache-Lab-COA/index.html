前排吐槽

这次的实验和以往有了比较大的不同。首先，从结构上看要复杂一些，有了不同的模块，更像是实现一个东西，而非过程式的实现得到特定输出的函数。当然，这也给测试代码带来了一些难度，同时增加了阅读框架代码的难度。同时，这也表明了计组这门课的考试难度不会简单，为即将到来的期末而担忧╮(╯▽╰)╭

知识回顾

  • Cache存在的目的是什么？
    • 处理器的处理速度远远超过了存储器提供信息的速度而产生了一个bar，Cache的作用就是在处理器与主存之间添加一个缓存，加快存取速度。
    • cache中保存的是主存中块的副本。
  • Cache的工作原理是什么？
    • CPU先向Cache索要数据（Check），如果找到则命中（Hit），如果不命中则在主存中寻找数据（Miss），将主存中的数据整块搬到Cache中，然后再从Cache中返回信息给CPU。
    • 可这样理解，编程作业也是这样实现的：CPU只与Cache交互，从Cache当中Fetch信息，如果命中则返回信息（的位置），不命中则在主存中寻找信息并替换到Cache中。
  • 如何知道Cache是否命中？
    • 我们已知数据的地址，那么我们可以推知它在Cache的标志位（tags）是怎么样表示的。在组相联映射中，还要先推得组号，在对应的组中匹配标志。同时，我们还需要注意的是有效位（ValidBit），这一点在课上没有提及，如果有效位置为0，则Cache是不被使用的，因此，在FIFO下，判断Cache是否被使用除了判断时间戳（TimeStamp）还需要判断有效位是否是1。
    • 因此，在标志位相匹配和有效位为1的情况下，Cache命中。
  • 时间局部性和空间局部性
    • 时间局部性：同一个数据在短时间内被使用多次；
    • 空间局部性：临近的数据在短时间被被连续使用；
  • 在替换的过程中，搬运的是块而不是字。
  • Cache之所以需要更多的操作却还能节省时间：
    • 是因为它有较高的命中率；
    • Cache的存取时间远小于到主存的存取时间；
  • Cache越大效果越好吗？
    • 增大Cache的大小可以提高命中率；
    • 但是也增大了成本和Cache的存取时间；
    • 不是，变好的幅度会越来越小。“局部性”，并不会牵扯到太大的范围，并且块越大块内寻找的成本就会越高。块太大了之后，带来的数据是局部性比较差的、比较老的数据。
  • 哪种映射方式更好？
    • 组相联是直接映射和全相联的一个Trade Off。
    • 当组相连的组大小为整个Cache时，映射模式本质上是全相联映射；当组大小为1时，本质上是直接映射；
  • Cache系统的组成？
  • 替换算法都有什么？分别有怎样的特点和好处？
    • LRU-最近最少用；
    • FIFO-先进先出；
      • 本次实验使用的主要算法：通过为Cache行添加上一个时间戳来标示CacheLine在Cache里面的存在时间，每次替换时间戳最大的那个。
    • LFU-使用频率最低；
    • Random-随机；
  • 写策略的异同？
    • 直写法（Write Through）
      • 替换的时候同时写主存和Cache；
      • 一改就要访问主存；
      • 提高了cache和主存的一致性；
    • 回写法（Write Back）
      • 只修改Cache。只有当主存有变化的时候才将Cache中被修改的信息写回主存；
      • 需要一个脏位来表示这一个块是否被调整过；如果是1就写回去，如果是0就替换掉；
      • 优点：减少对主存的操作到最小的程度；
  • CacheLine的行大小是不是越多越好？
    • 当行大小从很小变大的时候，命中率会相对提高——更多有用的信息会被搬到Cache中去；
    • 但当行大小过大，命中率就会降低
      • Cache中的行数会减少，从而导致更多的行替换；
      • 添加进去的字距离需要的字比较远，以至于小概率在短期内会被用到；
  • 多级Cache有什么好处？（亟待补充）

Cache的数据结构

/**
	 * Cache行，每行长度为(1+22+{@link Cache#LINE_SIZE_B})
	 */
private class CacheLine {
  // 有效位，标记该条数据是否有效
  boolean validBit = false;
  // 脏位，标记该条数据是否被修改
  boolean dirty = false;

  // 用于LRU和FIFO算法，记录该条数据时间戳
  Long timeStamp = -1L;

  // 标记，占位长度为()22位，有效长度取决于映射策略：
  // 直接映射: 12 位
  // 全关联映射: 22 位
  // (2^n)-路组关联映射: 22-(10-n) 位
  // 注意，tag在物理地址中用高位表示，如：直接映射(32位)=tag(12位)+行号(10位)+块内地址(10位)，
  // 那么对于值为0b1111的tag应该表示为0000000011110000000000，其中前12位为有效长度，
  // 因为测试平台的原因，我们无法使用4GB的内存，但是我们还是选择用32位地址线来寻址
  char[] tag = new char[22];

  // 数据
  char[] data = new char[LINE_SIZE_B];

  char[] getData() {
    return this.data;
  }
  char[] getTag() {
    return this.tag;
  }

  void update(char[] tagN, char[] input){
    validBit = true;
    timeStamp = 0L;
    System.arraycopy(tagN, 0, tag, 0, tagN.length);
    System.arraycopy(input, 0, data, 0, input.length);
  }

以上删去了无用的数据结构，这里的成员变量都已经在注释中解释的很清楚，不做过多解读了。主要说一下update，这里的update的主要目的是启用一个新的Cache，重置了它的有效位、时间戳以及内容。

故事要从$Cache$的读写开始说起。由于$Cache$需要进行分块读写，所以我们提供了一个helper函数，这个函数已经写好了，他做的事情就是分块读写，然后根据参数来判断究竟是读还是写。在helper中，我们调用了fetch函数，这个函数会检查$Cache$是否包含某一个特定的$block$，如果检查到了就会返回，没有检查到就会从内存中加载。因为我们只需要实现一种替换策略，所以这里你可以硬编码一个FIFO进去，对于映射策略来说，本质上只有一种映射策略。

private char[] helper(String eip, int len, char[] writeData){
  char[] data = new char[len];
  Transformer t = new Transformer();
  int addr =  Integer.parseInt(t.binaryToInt("0" + eip));
  int upperBound = addr + len;
  int index = 0;
  while (addr < upperBound) {
    int nextSegLen = LINE_SIZE_B - (addr % LINE_SIZE_B);
    if (addr + nextSegLen >= upperBound) {
      //为满足循环终止条件，补满nextSegLen
      nextSegLen = upperBound - addr;
    }
    int i=0;
    if(writeData == null){
      int rowNO = fetch(t.intToBinary(String.valueOf(addr)), nextSegLen);
      char[] cache_data = cache.get(rowNO).getData();
      while (i < nextSegLen) {
        data[index] = cache_data[addr % LINE_SIZE_B + i];
        index++;
        i++;
      }
    }
    else{
      int rowNO = fetch(t.intToBinary(String.valueOf(addr)), nextSegLen);
      char[] input = new char[LINE_SIZE_B];
      Arrays.fill(input, '0');
      while (i < nextSegLen) {
        input[addr % LINE_SIZE_B + i] = writeData[index];
        index++;
        i++;
      }
      writeStrategy.write(rowNO, input);
    }
    addr += nextSegLen;
  }
  return data;
}

helper用在read和write之中，减少了很大程度的重复代码；因为字符串操作的蛋疼性（手动熊猫擦汗），因此对addr需要一个神奇的操作。upperBound表示的是读写的终点。nextSegLen表示一行中还有多少剩余的位置。根据读写的数据量（即$len$的不同，很有可能会产生跨Cache行的读写，也很有可能产生不足一个Cache行的读写。

后面的代码都比较好理解，判断是否有内容需要写，从而决定是读还是写。如果是读的话，返回cache中的data。如果是写的话，通过写策略将Cache中写入内容。

以上是本次编程作业的一个主要逻辑单元。那么基于此开始总结本次的编程作业，首先是第一个我们参与到的函数——fetch

这是知识回顾中一个简单流程的实现。

框架代码中提供了一个处理前22位的函数，它非常的简单，就是把标志位+组号转化为整数类型。

public int getBlockNO(String addr) {
  Transformer t = new Transformer();
  return Integer.parseInt(t.binaryToInt("0" + addr.substring(0, 22)));
}

首先提取了块地址的有关信息——标志和组号。然后通过映射关系找到对应的Cache行（可能会不命中）。如果命中的话（这表明标志位和有效位均符合要求），返回相应的行号，流程结束！如果不命中的话……就有点麻烦了( ´▽` )ﾉ。我们需要通过替换策略，到主存中把需要的块替换到Cache中。

基于逻辑的顺序，首先我们来看映射策略中的映射如何实现的：

public int map(int blockNO) {
  //TODO
  int setNO = blockNO % SETS;
  int start = setNO * setSize;
  int end = (setNO + 1)*setSize;
  char[] tag = getTag(blockNO);
  return replacementStrategy.isHit(start, end, tag);
}

SETS表明了组数，setSize表明了组大小（全文均表示此意）。setNO表示组号，start表示起始的行号，end表示终点行号+1。接着在Cache中查找我们要找的Cache行是否存在我们要找的内容。

public int isHit(int start, int end, char[] addrTag) {
  //TODO
  for(int i=start; i<end; i++){
    if (Cache.getCache().isMatch(i, addrTag)) return i;
  }
  return -1;
}

public boolean isMatch(int lineNO, char[] tag){
  CacheLine cacheLine = this.cache.get(lineNO);
  
  if(cacheLine == null) return false;
  if (!cacheLine.validBit) return false;
  if (!Arrays.equals(cacheLine.tag, tag)) return false;
  
  return true;
}

isMatch提供了判断逻辑——即我们需要首先有cacheLine，其次有效位需要为true，最后需要标志位匹配；如果判断命中，我们返回相应的行号，如果判断不命中返回-1。

回到fetch，如果命中，返回行号。如果不命中，我们就需要从主存中搬来信息。

public int writeCache(int blockNO) {
    //TODO
    int setNO = blockNO % SETS;
    int start = setNO * setSize;
    int end = (setNO + 1)*setSize;
    char[] tag = getTag(blockNO);
    char[] memory =Memory.getMemory().read(t.intToBinary(String.valueOf(Cache.LINE_SIZE_B * blockNO)), Cache.LINE_SIZE_B);
    return replacementStrategy.Replace(start, end, tag, memory);
}

Memory.getMemory().read(t.intToBinary(String.valueOf(Cache.LINE_SIZE_B * blockNO)), Cache.LINE_SIZE_B);这么老长一串是从主存中读到的信息。

我的替换算法实现的比较复杂TAT，水平太低，并且肉眼可见的低效。不过还是先简单说说我自己的想法：为每个Cache行标记一个时间戳，没有被使用到的Cache行置为-1。一旦被使用到则时间戳变为0。在FIFO算法中，替换首先得组满了再进行替换，如果没满，则先使用空闲的。空闲的Cache行有两种，一种是没被使用过时间戳为-1，一种是有效位为false。

如果存在没被使用过的，则替换第一个没被使用过的free CacheLine。

如果满了，就在全部的CacheLine中寻找时间戳最大，在Cache中存在最久的那个，将其替换。

随后为组中全部非空Cache的时间戳增加1。

最后更新CacheLine。

public int Replace(int start, int end, char[] addrTag, char[] input) {
  //TODO
  //find free
  int index = 0;
  boolean isFull = true;
  for(int i=start; i<end; i++){
    if(Cache.getCache().getTimeStamp(i) == -1L || !Cache.getCache().getValid(i)){
      index = i;
      isFull = false;
      break;
    }
  }
  //Sort & Find the Oldest.
  if(isFull){
    long maxTime = Cache.getCache().getTimeStamp(start);
    index = start;
    for(int i=start; i<end; i++){
      if(maxTime < Cache.getCache().getTimeStamp(i)){
        maxTime = Cache.getCache().getTimeStamp(i);
        index = i;
      }
    }
  }
  //inc the TimeStamp
  for(int i=start; i<end; i++){
    long time = Cache.getCache().getTimeStamp(i);
    if(time >= 0L){
      Cache.getCache().incTimeStamp(i);
    } else {
      break;
    }
  }
  //inc the TimeStamp of the new CacheLine
  if(Cache.getCache().getTimeStamp(index)==-1L){
    Cache.getCache().incTimeStamp(index);
  }
  Cache.getCache().updateCacheLine(index, addrTag, input);
  return index;
}

那么关于“写”呢？

我们在知识回顾中提到了，本次实验所使用的回写法，只在我们对主存进行写操作时才把对Cache的修改写回主存。【读操作不进行写回，因此可能产生一致性的问题】

对Cache的写我们还是要看回helper。我们是通过WriteStrategy对Cache进行写操作的。

public void write(int rowNo, char[] input) {
  //TODO
  Cache.getCache().setDirty(rowNo, true);
  Cache.getCache().setData(rowNo, input);
}

首先，我们要把修改的那行Cache的脏位置为true，提醒主存在写操作（$altered）的时候回写。然后修改Cache里的内容，setData等函数不多赘述了。

public void write(String eip, int len, char []data){
  for(int i=0; i<1024; i++){
    if(Cache.getCache().getDirty(i)) {
      Cache.getCache().makeWB(i);
      Cache.getCache().setDirty(i, false);
    }
  }
  // 通知Cache缓存失效
  Cache.getCache().invalid(eip, len);
  // 更新数据
  for (int ptr=0; ptr<len; ptr++) {
    memory[Integer.parseInt(new Transformer().binaryToInt(eip)) + ptr] = data[ptr];
  }
}

写回的操作是这样的，先调用Cache，找到对应的Cache行，通过自身信息推导出需要写回的内存的地址，写入信息。

public void makeWB(int i){
  this.writeStrategy.writeBack(i);
}
public void writeBack(int rowNo) {
  //TODO
  Cache cache = Cache.getCache();
  String pAddr = mappingStrategy.getPAddr(rowNo);
  Memory.getMemory().writeBackM(pAddr, cache.getData(rowNo));
}

上面是写操作策略中的WriteBack，首先我们需要推出对应的物理地址pAddr。

public String getPAddr(int rowNo) {
  //TODO
  //
  int setNum = rowNo / setSize;
  String setStr = new Transformer().intToBinary(String.valueOf(setNum)).substring(32-countSets(SETS));
  char[] tag = Cache.getCache().getTag(rowNo);
  String tagStr = String.valueOf(tag).substring(0, 22-countSets(SETS));
  return tagStr + setStr + "0000000000";
}

public int countSets(int SETS){
  int ret = 0;
  SETS >>= 1;
  while(SETS != 0){
    ret++;
    SETS >>= 1;
  }
  return ret;
}

首先需要算出的是组数，这可以通过rowNo / setSize得到，比如组大小是4行，在第17行说明在第4组。（以0为起始）。然后算出Tag，它的长度是22-组数的长度。末尾补上十位0就是对应的物理地址。【为什么补上0就可以呢？因为Cache是按块操作的，以"0000000000"对齐的就是一块。】

然后就可以写入主存了！

public void writeBackM(String eip, char[] data){
  for (int ptr=0; ptr<1024; ptr++) {
    memory[Integer.parseInt(new Transformer().binaryToInt(eip)) + ptr] = data[ptr];
  }
}

一个常规的写操作。

主存的写操作也是有不少值得说的地方，除了判断是否有内容需要写回，将脏位恢复之外。还需要将重写的地方对应的Cache无效化。然后再更新数据。下面给出invalid但不做过多解释。

public void invalid(String sAddr, int len) {
  int from = getBlockNO(sAddr);
  Transformer t = new Transformer();
  int to = getBlockNO(t.intToBinary(String.valueOf(Integer.parseInt(t.binaryToInt("0" + sAddr)) + len - 1)));

  for (int blockNO=from; blockNO<=to; blockNO++) {
    int rowNO = mappingStrategy.map(blockNO);
    if (rowNO != -1) {
      cache.get(rowNO).timeStamp = -1L;
      cache.get(rowNO).validBit = false;
    }
  }
}
